## Plan: Azure Local Surveyor — Comprehensive Quality Overhaul (v2.0)

Complete overhaul. Resiliency and provisioning (fixed/thin) move to per-volume on the Volumes page. Each workload planner gets intelligence upgrades. AKS becomes multi-cluster. Arc services integrate into AKS. Volumes page is rebuilt with correct calculations. Reports get per-workload conditional tabs. Quick fixes for nav, docs, about page.

---

### Phase 1: Quick Global Fixes (parallel, no dependencies)

1. **Remove Home nav item** — Layout.tsx. Logo already navigates home.
2. **Update About page** — AboutPage.tsx. Add missing v1.5.0, v1.6.0, v1.7.0, v2.0 entries.
3. **Surface architecture docs** — DocsPage.tsx. Add links to overview.md and `engine-flow.md`.
4. **Verify References** — ReferencesPage.tsx. Audit all links, add any missing AKS/MABS/custom references.

---

### Phase 2: Store Schema & Type Foundation (*blocks everything else*)

Single Zustand migration v8→v9. All type changes in one shot:

- **VolumeSpec** → add `provisioning: 'fixed' | 'thin'`
- **Remove** `virtualMachines.resiliency`, `aks.resiliency`, `mabs.scratchResiliency`, `mabs.backupResiliency`
- **Keep** `mabs.internalMirror`, `sofs.internalMirror`, `advanced.defaultResiliency`
- **AKS** → `clusters: AksCluster[]` (name, control plane 1|3, workers, specs, PVC per cluster)
- **VMs** → `groups: VmStorageGroup[]` (name, VM count, specs per group)
- **SOFS** → add `volumeLayout: 'shared' | 'per-vm'`, `sofsOsDiskPerVmGB`
- **AVD** → drop `'s2d'` from storage location options
- **CustomWorkload** → drop `resiliency`
- **HardwareInputs** → drop `volumeProvisioning`

---

### Phase 3: Hardware Page Cleanup (*depends on 2*)

- Remove volume provisioning dropdown from HardwareForm.tsx. Provisioning is now per-volume.
- Everything else stays. Cache drives stay as informational (used by OEM presets). GPU, network, 3-tier: out of scope for v2.0.

---

### Phase 4: AVD Planner Intelligence (*depends on 2*)

- **Remove "Profiles on S2D"** option — not a viable production pattern.
- **Conditional behavior by storage location:**
  - **SOFS** → all profile fields active. SOFS sync active.
  - **Azure Files / External** → profile fields read-only/dimmed. "Estimated reference only — not included in cluster planning."
- **Separate data/temp disk** from profiles into its own "Session Host VM Storage" section.
- **Verify AVD session host VMs** are properly included in compute reporting.

---

### Phase 5: AKS Planner — Multi-Cluster (*depends on 2*)

- **Multi-cluster UI** like AVD host pools. Add/remove clusters, each with own config.
- **Remove** resiliency selector and `dataServicesTB` (Arc presets handle data services).
- **Engine** iterates `clusters[]` and aggregates totals.

---

### Phase 6: Arc Services → AKS Integration (*depends on 5*)

- Arc presets (SQL MI, IoT Ops, AI Foundry, Container Apps) **consume AKS worker capacity**, not additive.
- Preset storage flows as **PVCs within AKS**, not standalone volumes.
- **Enforce dependency:** AKS must be enabled if any Arc preset is enabled (blocking error, not just banner).
- Warn if preset demand exceeds total AKS worker capacity.

---

### Phase 7: SOFS Planner Overhaul (*depends on 2*)

- **Volume layout switch:** "Single shared volume" vs "One per SOFS VM"
  - Shared: 1 `SOFS-ProfileData` volume (size = `internalFootprintTB`)
  - Per-VM: N volumes (each `internalFootprintTB / N`)
- **OS disk volumes:** Add `sofsOsDiskPerVmGB` field. Generate OS disk volume(s), three-way-mirror default.
- **Only Azure Local host-level volumes** on Volumes page. Internal mirror is reflected in SIZE (already compounded), not as separate guest volumes.

---

### Phase 8: MABS Cleanup (*depends on 2*)

- **Remove** `scratchResiliency` and `backupResiliency` dropdowns. Keep `internalMirror`.
- 3 volumes still generated: Scratch, BackupData, OsDisk. Sizes account for internal mirror.
- **Provisioning defaults:** BackupData → `thin` (grows over retention), Scratch + OsDisk → `fixed`.
- Only Azure Local host-level volumes. No guest volumes on Volumes page.

---

### Phase 9: Virtual Machines — Storage Groups (*depends on 2*)

- **Storage groups:** Default 1 group (backward compatible). User can add groups.
- Each group: name, VM count, vCPUs/VM, RAM/VM, storage/VM.
- Each group → separate volume suggestion (`VM-{name}`).
- Use case: 10 prod VMs three-way + 5 dev VMs two-way → configure on Volumes page.

---

### Phase 10: Custom Workloads (*depends on 2*)

- **Remove `resiliency`** from template and UI (per-volume now).
- **Wire `internalMirrorFactor`** — currently stored but NOT used in engine. Apply: `storageTB × internalMirrorFactor` for volume size.
- Per custom workload: OS disk volume + data volume suggestions.
- Validate imported JSON against schema.

---

### Phase 11: Workload Planner Totals (*depends on 5, 6, 9*)

- Fix vCPU/Memory aggregation bars at top of WorkloadPlanner.tsx.
- After Phase 6, Arc presets consume AKS capacity → remove double-counting.
- Verify each workload's contribution is correct after all schema changes.

### Phase 12: Volumes Page Overhaul (*the big one, depends on all workload phases*)

#### 12A. Quick Start Reference — COMPLETE REWRITE
- **Pure hardware reference.** Zero workload dependencies. Zero FAIL status.
- **Two rows:** Three-Way Mirror and Two-Way Mirror.
- **Columns:** Resiliency | Volume Count | Calculator Size (TB) | WAC Size (TiB) | WAC Size (GiB) | Pool Footprint (TB)
- **Remove:** "Fits?" column, "Scenario" column, FAIL/PASS banners.
- **Fix rounding:** After TiB floor, subtract 1 GiB safety margin to guarantee fit. This is a reference, not exact.
- Keep PowerShell copy-to-clipboard script.

#### 12B. Suggested Volumes (Generic + Workload modes)
- Add per-suggestion **resiliency dropdown** (three-way, two-way, dual-parity). Default from `advanced.defaultResiliency`.
- Add per-suggestion **provisioning dropdown** (fixed / thin). Default `fixed` (except MABS-BackupData → `thin`).
- When user clicks Add/Add All → volumes carry chosen resiliency + provisioning to planned table.
- Workload mode: Intelligence from planners (SOFS layout, VM groups, AKS clusters) drives suggestion count and sizes. Only Azure Local host-level volumes.

#### 12C. Planned Volumes Table
- **New columns:** Name | Resiliency | Provisioning | Planned (TiB) | New-Volume GiB | Pool Footprint (TB) | Edit/Delete
- Inline editor gains provisioning dropdown.

#### 12D. Storage Utilization Bar — FIX
- Verify `totalPoolConsumptionTB = sum(plannedSizeTB / resiliencyFactor)` per volume. Per-volume resiliency, NOT global.
- **Thin provisioning awareness:** Show logical allocation vs physical pool. Thin volumes can over-provision.
- Tooltip: "Pool consumption relative to available capacity after reserves."

#### 12E. Volume Health Check — OVERHAUL
- **Remove compute checks** (vCPU, memory) — those belong in Compute Report.
- **Always show all checks** — green pass included, not just failures.
- **Thin provisioning callout:** If thin-provisioned and logical total exceeds capacity → info (blue): "You are thin-provisioned by X TB over physical. Expected with thin provisioning — requires monitoring." NOT an error.
- **Fixed volumes:** If total exceeds capacity → ERROR (red).
- **Per-volume detail:** Every volume shows resiliency, provisioning, size, pool footprint, status.

#### 12F. Fix Wrong Calculations
- Audit `computeVolumeSummary()` end to end.
- Utilization bar denominator should be `availableForVolumesTB` (raw pool space), NOT `effectiveUsableTB`.
- Resiliency factor MUST be per-volume (already is in code — verify no bugs).
- Run test scenarios to validate.

### Decisions
1. Resiliency → per-volume on Volumes page. `advanced.defaultResiliency` = default for suggestions.
2. Provisioning → per-volume. Fixed or thin independently.
3. Arc presets → consume AKS worker capacity (not additive). Storage as AKS PVCs.
4. SOFS/MABS volumes → only Azure Local host-level CSVs. Guest storage reflected in SIZE via mirror factor.
5. Quick Start → pure hardware reference, no FAIL.
6. Health check split → volume checks on Volumes page, compute checks in report.
7. Thin provisioning → over-provisioning is info, not error.

### Scope Exclusions
- GPU, network, 3-tier storage, cache engine integration, host OS breakdown, stretched cluster — all future.
