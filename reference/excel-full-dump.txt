[P2]: WORKLOAD AND STORAGE CALCULATOR — HOW TO USE
[P2]: Azure Local / Storage Spaces Direct — Capacity Planning Tool
[P2]: This workbook helps you plan storage for an Azure Local cluster using Storage Spaces Direct (S2D). Enter your hardware specs and planned workloads — the calculator tells you how much usable storage you get and whether everything fits. You do NOT need to be a storage expert.
[P2]: QUICK START — 3 STEPS
[P3]: STEP 1  |  [P4]: Go to 'Hardware Inputs'. Use the DROPDOWN MENUS to enter: node count, drive type, drives per node, and drive size. The calculator IMMEDIATELY shows your available storage in a big green number.
[P3]: STEP 2  |  [P4]: Go to 'Workload Planner'. Choose your scenarios: AKS, Infrastructure VMs, Dev/Test, Backup, or Custom. For AVD, go to the 'AVD Planning' tab — all AVD inputs live there. Running counter shows remaining capacity.
[P3]: STEP 3  |  [P4]: Review 'Capacity Report' and 'Volume Detail'. Print or share these with your team or customer. The report explains all the numbers and includes key concepts if anyone asks questions.
[P2]: COLOR GUIDE
[P3]: Blue on Yellow  |  [P4]: YOUR INPUT — only cells you edit. All have dropdowns or accept numbers.
[P3]: Green  |  [P4]: GOOD NEWS — available capacity, passing checks. Bigger green number = better.
[P3]: Blue on Light Blue  |  [P4]: CALCULATED — auto-calculated values. Do not edit.
[P3]: Red / Pink  |  [P4]: WARNING — footprint numbers, overcommit alerts, capacity exceeded.
[P3]: Orange  |  [P4]: CAUTION — reserve capacity, infrastructure overhead.
[P3]: Gray  |  [P4]: LABELS — descriptions explaining what each row means.
[P2]: TABS IN THIS WORKBOOK

[P2]: COMMON QUESTIONS
[P3]: Why does Windows show different numbers?  |  [P4]: Calculator uses TB (decimal). Windows uses TiB (binary). 1 TB = 0.909 TiB. Both are correct. Row 26 on Hardware Inputs shows TiB equivalent.
[P3]: Why is so much capacity 'lost'?  |  [P4]: Three-way mirror = 3 copies = 33% efficiency. This is WHY data survives 2 failures. Two-way = 50%. Dual Parity = 50-80% at 4+ nodes.
[P3]: Why can't I use all my pool?  |  [P4]: Three deductions: Drive overhead (~8%), Reserve (~1 drive/node for auto-repair), Infrastructure volume (~250 GB for Azure Local system).
[P3]: What if workloads don't fit?  |  [P4]: Options: fewer VMs, Two-Way instead of Three-Way where OK, more/smaller drives (see Drive Layout Comparison tab), more nodes, Dual Parity for archives.
[P3]: Should I use thin provisioning?  |  [P4]: Production: NO, use Fixed. Thin risks pool exhaustion = VMs crash. OK for dev/test and FSLogix with monitoring. See Thin Provisioning tab.
[P3]: NVMe+SSD vs All-NVMe?  |  [P4]: NVMe+SSD: NVMe becomes CACHE (not capacity). Only SSD counts. All-NVMe: all drives are capacity. Most common for new Azure Local deployments.
[P3]: Can I add drives/nodes later?  |  [P4]: Yes. New drives auto-join pool. Adding 3rd node unlocks three-way mirror. But sizing correctly upfront is simpler than expanding later.
[P3]: What does N+1 compute mean?  |  [P4]: Size VMs to fit on N-1 nodes. If 1 node goes down, all VMs must failover to remaining nodes. This is CPU/RAM, separate from storage.
[P2]: TIPS
[P2]: 1.  |  [P3]: Start with Hardware Inputs before planning workloads. Know what you have before planning what to put on it.
[P2]: 2.  |  [P3]: Use dropdown menus — don't type resiliency types manually. Typos break formulas.
[P2]: 3.  |  [P3]: Include ALL disks when estimating VM sizes (OS + data disks). Typical Windows Server = 80 GB OS + 100 GB data = 180 GB.
[P2]: 4.  |  [P3]: For FSLogix, start at 5 GB/user if no data. Measure after 30 days and adjust.
[P2]: 5.  |  [P3]: Keep pool utilization below 70%. Above 70% = limited rebuild headroom. Above 90% = danger zone.
[P2]: 6.  |  [P3]: Consider Dual Parity for backup/archive on 4+ node clusters: 50-80% efficiency vs 33% for three-way mirror.
[P2]: 7.  |  [P3]: Print the Capacity Report tab for project documentation.
[P2]: 8.  |  [P3]: This calculator uses TB. Windows uses TiB. Numbers WILL look different. Both are correct.

[P2]: DEFAULT: USE FIXED (THICK)

[P2]: Pool Exhaustion  |  [P3]: Pool full = volumes OFFLINE = VMs crash. HARD STOP.
[P2]: Silent Overcommit  |  [P3]: Logical can exceed physical. No hard block.
[P2]: Growth Spikes  |  [P3]: Updates, snapshots consume GBs with no warning.
[P2]: No Reclaim  |  [P3]: Stretched clusters: TRIM disabled. Volumes only grow.
[P2]: Double-Thin  |  [P3]: Thin S2D + thin VHDX = two overcommit layers.
[P2]: Rebuild Blocked  |  [P3]: No pool space = rebuild fails = data at risk.
